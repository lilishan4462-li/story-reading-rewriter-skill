# 对话框平台使用说明

本文件说明豆包、DeepSeek、Kimi、通义、普通 ChatGPT 等对话框软件使用 Lite 版时的差异。

## 总原则

对话框模型不是完整 Skill 运行环境。

它们通常不能长期保存文件结构，也不能像 Codex、Claude Code、Cursor 那样稳定维护项目目录。因此 Lite 版采用：

- 单文件规则包；
- 模型主动主持流程；
- 每轮项目状态卡；
- 分批输入材料；
- 单章生成、单章质检、单章修复。

## 豆包

适合上传文件后进行长文本对话，但测试中发现两个风险：

1. 文件过长时，豆包可能只读取前 70% 到 85%。
2. 它可能把原版电子书先压缩成“全文内容总结”，然后基于总结继续跑。
3. 它容易把“追剧式改写”理解成通俗解说专栏，而不是知识小说。

建议：

- 先上传 `story-reading-rewriter-lite.md`；
- 再上传书籍、目录或章节；
- 如果豆包提示“只阅读了前 82%”，不要继续正式改写，先补充当前章节正文；
- 如果豆包只输出“全文内容总结”，要求它停止总结，先做电子书读取验收；
- 如果它输出“上期回顾、本期看点、互动思考题、下集预告”，立刻要求它回到知识小说模式；
- 每次输出必须检查是否包含“知识揭底、质量闸门、项目状态卡”。

豆包纠偏提示：

```text
停止。你刚才偏离了追剧式知识小说模式，写成了通俗解说稿。
请回到 Lite 流程：
电子书读取验收 -> 材料判断 -> 内容大纲 -> 章节计划 -> 单章小说正文 -> 知识揭底 -> 质量闸门 -> 项目状态卡。
如果当前看到的只是平台压缩后的全文总结，请不要写正式章节，请先确认是否读到了目录和第 1 个正文章节。
```

## DeepSeek

适合文本推理、结构分析和中文长文改写。

建议：

- 如果网页端文件能力有限，就直接粘贴目录和当前章节；
- 每次续写前粘贴项目状态卡；
- 不要一次要求生成太多章。

## Kimi

适合处理较长材料。

建议：

- 可以先让它读取目录和多章内容；
- 仍然要求它先输出内容大纲；
- 如果上下文很长，要求它定期压缩状态卡。

## 通义

适合中文材料处理和常规文档对话。

建议：

- 用 Lite 规则包固定流程；
- 分批推进章节；
- 对输出进行质量闸门检查，避免变成读书笔记。

## 普通 ChatGPT 对话框

适合使用自定义 GPT、项目文件或普通上传文件。

建议：

- 如果支持项目文件，把 Lite 规则包作为项目知识；
- 如果只是普通对话，先粘贴 Lite 规则包，再上传材料；
- 每次长对话后保留项目状态卡。

## 常见失败

### 失败 1：模型直接总结原书

修正提示：

```text
停止总结。请按追剧式读书改写器 Lite 流程，先建立内容大纲，再进行小说化改写。
```

### 失败 2：模型每章换一个故事

修正提示：

```text
停止。全书必须使用同一个主角、同一个世界、同一条主线。
请根据当前项目状态卡做定向修复，不要重开设定。
```

### 失败 3：模型把人物写成老师

修正提示：

```text
停止讲课式写法。
请把概念改成场景中的冲突、选择、代价和后果。
人物不能像老师一样解释概念。
```

### 失败 4：模型忘记前文

修正提示：

```text
下面是最近一次项目状态卡。
请先恢复上下文，复述内容大纲、已完成章节和当前未解钩子，再继续。
```

### 失败 5：模型写成通俗解说专栏

修正提示：

```text
停止。当前输出不是知识小说，而是通俗解说稿。
请删除“上期回顾、本期看点、互动思考题、下集预告”等栏目。
改为有主角、有场景、有冲突、有代价的小说正文。
正文后再写知识揭底、质量闸门和项目状态卡。
```
